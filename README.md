## About the Project

**JBC Athenaeum** is a modern academic-resource platform for students and faculty of **Jana Bhawana Campus**.

It is built with **React 19, TypeScript, Vite, Tailwind CSS, Supabase, Cloudflare Pages, and Capacitor 8**. The platform is designed to help students discover, contribute, review, and access verified academic materials through a secure and moderated workflow.

The project supports both web and native mobile delivery while sharing the same React and Supabase application core.

## Local Development

### Requirements

Before starting, install:

* Node.js
* npm
* Git
* A Supabase project
* A modern Chromium-based browser

### Installation

```sh
git clone https://github.com/SanjelNirmal/JBCATHENAEUM.git
cd JBCATHENAEUM

npm install
cp frontend/.env.example frontend/.env.local
npm run dev
```

The development server will start using the workspace configuration defined in the repository.

### Environment Configuration

Set only the public Supabase project values in:

```text
frontend/.env.local
```

Example:

```env
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-public-anonymous-key
```

> [!CAUTION]
> Never place a Supabase service-role key, database password, Cloudflare API token, SMTP password, signing secret, or other privileged credential in a `VITE_` variable.

All variables prefixed with `VITE_` are embedded into the frontend bundle and must be considered public.

## Optional Project Support

JBC Athenaeum academic resources remain free.

The footer's **Buy me a coffee** action is an optional bank-transfer prompt. It is never used as:

* A document access check
* A download requirement
* A paid-resource gate
* A subscription mechanism
* A user-ranking system
* A moderation advantage

Bank display details are centralized in:

```text
frontend/src/config/payment.ts
```

To enable the payment QR display, add the verified NMB QR image at:

```text
frontend/public/payment/nmb-payment-qr.png
```

When the image is missing, the modal safely falls back to the configured bank details.

The application does not collect:

* Payment receipts
* Banking credentials
* Transaction screenshots
* Transaction references
* Card information
* Mobile-banking passwords

> [!IMPORTANT]
> The native application does not display the optional support prompt. Paid-content and external-payment controls remain disabled until Apple and Google platform-policy review is complete.

## Android and iOS

Capacitor belongs to the `frontend` workspace.

```sh
cd frontend
npm run cap:sync
npx cap open android
npx cap open ios
```

### Native Application Identifier

```text
np.com.nirmalsanjel.jbcathenaeum
```

### Supported Deep-Link Shapes

```text
jbcathenaeum://auth/callback
jbcathenaeum://payment/return
```

Deep links are navigation inputs only. Authentication must still be validated through Supabase using a valid server-verifiable authentication flow.

A payment-return URL must never:

* Unlock protected content
* Mark a transfer as successful
* Assign a paid role
* Approve an account
* Bypass server-side verification

### iOS Personal Team Development

For local iOS builds using a Personal Team:

* Keep automatic signing enabled
* Do not add the `Associated Domains` capability
* Use only capabilities supported by the active provisioning profile
* Verify the bundle identifier before archiving

The `Associated Domains` capability requires an appropriate Apple Developer account and matching provisioning profile.

See:

[Mobile release runbook](docs/MOBILE_RELEASE.md)

The runbook covers:

* Android and iOS prerequisites
* Signing
* Deep-link association
* Supabase redirect URLs
* Privacy declarations
* Payment-policy constraints
* Honest test status
* Store-release blockers

## Deployment Status and Limitations

The repository is locally hardened, but repository readiness does not prove that every live service is fully configured.

The following owner-controlled systems must be verified separately:

* Live Supabase database configuration
* Production Row Level Security policies
* Storage bucket permissions
* Deployed Edge Functions
* Cloudflare Pages environment variables
* Authentication redirect URLs
* SMTP configuration
* Monitoring providers
* Audit-log retention
* Backup schedules
* Recovery testing
* Campus governance approval
* Mobile store compliance
* Institutional data-handling policies

> [!WARNING]
> Do not describe the platform as production-ready solely because the local build and test suite pass.

A limited campus beta should begin only after the relevant production, security, governance, monitoring, and backup gates are verified.

## Security Principles

JBC Athenaeum follows these core principles:

| Principle                  | Implementation Direction                                                   |
| :------------------------- | :------------------------------------------------------------------------- |
| Least privilege            | Users receive only the permissions required by their role                  |
| Server-side authorization  | Sensitive operations are never trusted to frontend checks alone            |
| Private-by-default storage | New PDF files begin in a private quarantine bucket                         |
| Signed access              | Protected documents use signed or policy-controlled access                 |
| Strong moderation          | Sensitive moderation actions require MFA/AAL2                              |
| Auditable changes          | Important administrative actions generate audit records                    |
| Safe secrets               | Privileged credentials remain outside frontend bundles                     |
| Explicit verification      | Payment returns, deep links, and callbacks never grant trust automatically |
| Controlled deletion        | Resource deletion follows reviewed and traceable workflows                 |
| Data minimization          | The platform avoids collecting unnecessary payment and banking information |

## Contributing

Contributions should preserve the platform's academic, security, accessibility, and moderation requirements.

Before submitting a pull request:

1. Create a focused branch.
2. Keep secrets out of commits.
3. Follow the existing TypeScript and component conventions.
4. Add or update tests for changed behavior.
5. Verify keyboard navigation and responsive layouts.
6. Run all quality gates.
7. Document migration or deployment changes.
8. Avoid weakening RLS, MFA, storage, or moderation controls.

Example workflow:

```sh
git checkout -b feature/resource-improvement
git add .
git commit -m "feat: improve resource filtering"
git push origin feature/resource-improvement
```

Then open a pull request with:

* A clear summary
* Screenshots for interface changes
* Test results
* Migration notes
* Security implications
* Deployment considerations

## Project Governance

JBC Athenaeum is an independent academic-resource platform created for the Jana Bhawana Campus community.

The platform must not imply formal institutional endorsement unless such approval has been explicitly granted.

Academic materials should be:

* Relevant to the intended program or subject
* Legally shareable
* Properly attributed
* Free from malicious content
* Reviewed before publication
* Removed when a valid legal or institutional request is approved

See the campus governance documentation for operational responsibilities and approval requirements.
